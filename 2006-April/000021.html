For nice projects :)
> First, just a short explanation about the server side of OPDB.
> We use a MySQL server on BerliOS, and query scripts through which the
> browser extensions can send queries in Online mode. Then there's the option
> (actually it's default) to download the sites list as an XML file. Then the
> client queries the XML file instead.
Ah didn't now that, but using a server part will risk to make berlios down if 
there are too many users :) (actually i use online mode)

> > So i wanted to look a bit more, firstly the database format and i saw....
> > An xml file!
> > For the moment there are only 3 entries, so it's pretty little so it
> > doesn't take time to parse or to download, but you may get thousands
> > (million?) entries, so it will quickly become a nightmare to parse:I
> > tried to make a 10MB database and it took 10s for each time, even if
> > there was "many" requests to do, it's still too much for this application
> > (if it was 2s with blocking events and not cpu taking why not, but it's
> > mainly cpu and mem usage...).
>
> When we reach a larger number of entries, we would probably already have
> used libbz2 to compress the XML file, so the download time would be
> reduced. But as to the performance issue you're mentioning...
You may try to make a big database with "random" values, maybe it's quicker 
with SAX than with the DOM tree I used, but i think it will still take too 
many ressources for that

> > The alternative i use against XML is sqlite,it's less flexible than XML,
> > but for the opdb, i don't think there is any flexibility needed.
> > A rightly built sqlite DB (with right indexes), is pretty quick compared
> > to XML (maximum 1s for same requests)
>
> This is interesting... Though I have never used sqlite. I'd guess that
> since it's a database it's definitely faster than parsing an XML file...
> But we need to bring the site list from the server to the user. How do we
> do that? Right now we just have an XML file and we parse it directly. If
> you say we should avoid XML files, how would the client download the whole
> site list and update the local database?
Didn't thaught at this, but I can propose some ideas for that:
-Take the binary format of the db as generated by sqlite (i don't really like 
that but it can work)
-An alternative of that: take the xdelta and each day/hour (or what else) 
generate the delta to make it lighter to download
-Download the "dumped" version (the SQL queries so)
-An alternate: Only download "dumped" differences between the last 
hour/day/whatelse

Notice that i propose sqlite, because I know it "well" (I know how to use the 
C API at least), but there may be other solutions which will works well too, 
if they can give enough performances (and aren't too hard to code...)
